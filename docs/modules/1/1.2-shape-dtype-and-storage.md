# Module 1.2 — Shape, Dtype and Storage

> *A tensor is not its data. It is an interpretation of storage.*

---

# Objective

* explain the role of storage, shape, dtype, and strides;
* understand how a runtime locates tensor elements;
* explain why many tensor operations can be implemented without copying memory;

# Learning Level 1 — Intuition

Consider the tensor:

```text
shape = (2,3)

1 2 3
4 5 6
```

Although it appears as a matrix, memory does not contain rows or columns.

Instead, memory stores a linear sequence of values.

```text
1 2 3 4 5 6
```

The matrix exists only because the runtime interprets that storage using metadata.

---

# Memory Is One-Dimensional

Memory is fundamentally linear.

It contains addresses and bytes.

Concepts such as rows, columns, channels, batches, and embeddings do not exist in memory itself.

They emerge from the interpretation performed by the tensor abstraction.

This distinction between physical storage and logical representation is one of the central ideas of AI Mastery.

---

# Storage

Storage represents the physical memory containing tensor data.

Conceptually, it is simply a contiguous sequence of bytes.

The storage itself has no notion of dimensions or semantic meaning.

Different tensors may reference the same storage while exposing different logical views.

---

# Shape

Shape specifies how storage should be divided into logical dimensions.

Example:

```text
shape = (2,3)
```

The runtime interprets the same storage as:

```text
1 2 3
4 5 6
```

Changing only the shape changes the logical interpretation without necessarily changing the underlying memory.

---

# Dtype

Dtype specifies how each element stored in memory should be interpreted.

It determines:

* element size;
* numerical representation;
* precision;
* valid value range.

For example:

```text
float32 → 4 bytes
float64 → 8 bytes
int8     → 1 byte
```

Without dtype, the runtime cannot correctly interpret the bytes stored in memory.

---

# Strides

Strides describe how memory should be traversed.

Each stride specifies how many elements must be skipped when incrementing an index along a particular axis.

For a contiguous tensor:

```text
shape   = (2,3)
strides = (3,1)
```

This means:

* moving one row skips three elements;
* moving one column skips one element.

Strides map logical coordinates to physical storage.

---

# Tensor Views

Different tensors may share the same storage.

Example:

```text
Storage

1 2 3 4 5 6
```

Tensor A:

```text
shape   = (2,3)
strides = (3,1)
```

Tensor B:

```text
shape   = (3,2)
strides = (1,3)
```

Both tensors reference identical storage.

Only their interpretation differs.

This makes operations such as transpose possible without copying memory.

---

# Metadata-Only Operations

Many tensor operations modify only metadata.

Examples include:

* reshape;
* transpose;
* views.

When possible, these operations avoid copying storage, making them extremely efficient.

Copying memory is generally avoided unless required by the desired layout.

---

# Runtime Perspective

When evaluating an expression such as:

```text
tensor[i][j]
```

the runtime conceptually performs the following steps:

1. Validate the indices using the shape.
2. Compute the logical offset using the strides.
3. Convert the offset into a byte offset using the dtype.
4. Access the corresponding location within the storage.

The tensor itself does not contain a multidimensional array.

It contains enough metadata to reconstruct one.

---

# Tensor API Design

The minimal tensor abstraction should expose the concepts required to represent any tensor.

Conceptually:

```text
Tensor
├── Storage
├── Shape
├── Dtype
└── Strides
```

A primitive constructor should accept these concepts directly, while higher-level constructors may derive some metadata automatically.

Convenience APIs such as:

* `from_list()`
* `zeros()`
* `ones()`
* `random()`

should be viewed as helpers built on top of the primitive abstraction rather than independent concepts.

---

# Tensor Invariants

A valid tensor should satisfy several fundamental invariants.

Examples include:

* shape and strides must describe the same number of dimensions;
* the tensor must be representable within its referenced storage;
* strides must correctly map every logical index to a valid storage location;
* metadata must remain internally consistent.

Future modules will introduce additional invariants as views, slicing, offsets, and shared storage become more sophisticated.

---

# Engineering Insight

A tensor should not be viewed as a container of data.

Instead, it should be viewed as an object that interprets storage.

This separation enables:

* storage sharing;
* zero-copy views;
* efficient reshaping;
* runtime flexibility;
* generic tensor operators.

The same architectural principle appears repeatedly throughout modern AI infrastructure.
