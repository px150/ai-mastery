# Views

A view is a new tensor that shares the same storage while exposing different metadata.

Unlike copying operations, creating a view does not allocate new memory.

Instead, multiple tensors coexist over the same storage.

---

# Metadata-Only Transformations

Some tensor operations do not require moving data.

Instead, they create a new tensor by modifying metadata only.

Matrix transpose is the first example.

The implementation:

* preserves storage;
* preserves dtype;
* swaps shape;
* swaps strides.

No storage allocation occurs.

---

# Memory Aliasing

Because views share storage, modifying data through one tensor is immediately observable from every other tensor referencing the same storage.

Aliasing is therefore a consequence of shared storage rather than a special runtime feature.

---

# Why Transpose Works Without Copying

Transpose changes how tensor coordinates map to storage.

The physical memory remains unchanged.

Only the interpretation of each axis changes.

Since strides describe movement along tensor axes, swapping tensor axes naturally requires swapping their corresponding strides.

---

# The Limits of Metadata

Not every tensor transformation can be represented by changing metadata alone.

Some layouts cannot be described using only:

* shape;
* strides.

When the logical traversal of tensor elements no longer corresponds to a regular stride pattern, the runtime must allocate new storage and copy the data.

This explains why some reshape operations are metadata-only while others require materialization.

---

# Design Decisions

The runtime intentionally postpones introducing `storage_offset`.

Current implementation supports views that begin at the first element of storage.

Features such as slicing require additional metadata and will be introduced only when they become necessary.

# Tensor Contiguity

A tensor is **contiguous** when its logical traversal of storage matches the canonical contiguous memory layout for its shape.

Formally:

```python
tensor.strides == compute_contiguous_strides(tensor.shape)
```

Contiguity is therefore determined entirely by tensor metadata rather than by the storage itself.

---

# Physical Memory vs Logical Layout

Storage contains only a linear sequence of elements.

Tensor metadata defines how those elements are interpreted as a multidimensional object.

As a result, two tensors may share exactly the same storage while exposing completely different logical layouts.

Understanding this distinction is fundamental for reasoning about tensor runtimes.

---

# Contiguity Is a Property of the Tensor

It is important to distinguish between the storage and the tensor itself.

Storage is always a contiguous one-dimensional block of memory.

Contiguity instead describes whether the tensor's logical iteration order matches the physical order of elements in that storage.

A transpose therefore does **not** make memory fragmented.

Instead, it changes the order in which tensor elements are visited.

This distinction is fundamental throughout modern tensor runtimes.

---

# Why Transpose Is Not Contiguous

Transpose preserves storage but changes the mapping between tensor coordinates and storage offsets.

Although every element remains in the same physical location, the logical traversal order no longer matches the canonical contiguous layout for the new shape.

The tensor therefore becomes non-contiguous even though no data has been copied.

---

# Why Non-Contiguous Access Is Slower

Modern CPUs and GPUs are optimized for sequential memory access.

Memory is transferred in fixed-size cache lines rather than individual elements.

When elements are visited sequentially, nearly every loaded cache line is fully utilized.

For example:

```
0 → 1 → 2 → 3 → ...
```

Each cache line supplies many consecutive elements that will soon be used.

A transposed tensor often requires visiting memory with large strides:

```
0 → 10000 → 20000 → ...
```

Although this pattern is perfectly regular, each access may require loading a different cache line while using only a small fraction of its contents.

The hardware therefore performs significantly more memory transfers to process the same number of tensor elements.

---

# Why Contiguity Matters

Many tensor operations assume that memory can be traversed sequentially.

Examples include:

* reshape;
* flatten;
* vectorized kernels;
* BLAS routines;
* GPU memory transfers.

When a tensor is contiguous, these operations can traverse memory sequentially, allowing modern hardware to fully exploit cache locality, hardware prefetching and vectorized instructions.

When it is not, the runtime must either follow the tensor strides or materialize a new contiguous storage.

Following the strides preserves memory but may reduce performance, while materializing a contiguous copy increases memory usage but enables faster subsequent computation.

---

# Small Tensors Can Hide the Cost

For very small tensors, transpose may appear to have no performance penalty.

If the entire storage fits inside a few cache lines, every element is already available after the first memory access.

The performance differences become significant only when tensors are large enough that cache capacity becomes a limiting factor.

For this reason, tensor contiguity is primarily a performance concern for large numerical workloads rather than small educational examples.

---

# Why Materializing a Contiguous Copy Can Be Faster

Creating a contiguous copy requires allocating new storage and copying every element once.

This initial cost may seem unnecessary.

However, many subsequent operations—such as matrix multiplication, convolutions, reductions and vectorized kernels—benefit enormously from sequential memory traversal.

In these situations, paying the cost of a single copy can be substantially cheaper than repeatedly operating on a non-contiguous layout.

Tensor runtimes therefore postpone copying for as long as possible and materialize contiguous storage only when required or when doing so enables significantly more efficient execution.

---

# Engineering Insight

Tensor contiguity is not a correctness requirement.

It is a performance property.

A non-contiguous tensor represents exactly the same mathematical object as a contiguous tensor.

The difference lies entirely in how efficiently modern hardware can traverse its memory.

---

# Runtime Design

The runtime exposes tensor contiguity through:

* `Tensor.is_contiguous()`

Contiguity is intentionally computed on demand instead of being stored as runtime state.

This follows the repository principle of avoiding redundant derived state.

## Reshape

`reshape` changes the logical shape of a tensor while preserving the represented data.

Like `transpose`, it attempts to reuse the existing storage whenever possible.

However, unlike `transpose`, a reshape is not always representable through metadata alone.

A metadata-only reshape is possible only when:

* the total number of represented elements remains unchanged;
* the tensor is contiguous.

When these conditions hold, the runtime creates a new tensor that:

* shares the same storage;
* preserves the dtype;
* exposes the new shape;
* computes new contiguous strides for that shape.

No storage allocation occurs.

---

# Materialization and `contiguous()`

A non-contiguous tensor cannot always be reshaped by modifying metadata alone.

After operations such as transpose, the logical traversal order no longer matches the canonical contiguous layout required by the new shape.

In these situations, the runtime first materializes a contiguous copy.

This process:

* allocates new storage;
* copies elements in logical tensor order;
* constructs a new contiguous tensor.

Once the tensor becomes contiguous, reshape once again becomes a metadata-only transformation.

This separation reflects an important runtime principle:

Metadata transformations are preferred whenever possible.

Memory allocation is postponed until the existing metadata can no longer represent the desired logical layout.

Modern tensor runtimes attempt to avoid copying memory whenever possible.
